# Native packaging (Capacitor)

Gravy is a Vite/React PWA. To get real native capabilities (push, biometrics,
camera, store distribution) the plan (BACKLOG.md, Epic 10) is to **wrap the
existing web build in a thin [Capacitor](https://capacitorjs.com) native shell**
— no rewrite of `src/`. This document is the output of the initial packaging
spike: how the wrap is wired, how to build/run it, what's been verified, and
what's deliberately left for later.

## What the spike established

- `@capacitor/core` + `@capacitor/cli` + `@capacitor/android` are installed and
  `capacitor.config.ts` points the shell at the production web build (`dist/`).
- A dedicated **native build mode** (`CAPACITOR_BUILD=true`, see
  `vite.config.ts`) produces a `dist/` that actually works inside a native
  shell, where the web (GitHub Pages) build does not — see "Why a separate
  build mode" below.
- The Android platform is scaffolded (`android/`) and `cap sync` copies the
  built web app into it. This is the wrap working end-to-end up to the point of
  a Gradle/Xcode compile.

## What is *not* done here (follow-ups, see Epic 10)

- **No APK/IPA is built.** Compiling needs the Android SDK (Android Studio) and,
  for iOS, macOS + Xcode — neither is available in CI today. The spike stops at
  `cap sync`; the next step is opening the project in Android Studio / Xcode on a
  real dev machine.
- **iOS platform not added.** `npx cap add ios` requires macOS (CocoaPods). Add
  it on a Mac with the same one-liner; everything else (config, native build
  mode) already applies.
- **No native plugins wired yet** — push (APNs/FCM), biometrics, haptics,
  camera, widgets are all separate Epic 10 items that build on this wrap.
- **Signing, store accounts, Fastlane/CI for native builds, OTA-update policy,
  crash reporting, device-matrix testing** — all still open (Epic 10).

## Prerequisites to actually build

- Android: Android Studio (bundles the SDK + a JDK). Open `android/` in Android
  Studio, or set `ANDROID_HOME` and use the Gradle wrapper.
- iOS: macOS, Xcode, and CocoaPods.

## Workflow

```bash
# Build the web app in native mode AND copy it into the native project(s):
npm run cap:sync          # = npm run build:native (CAPACITOR_BUILD=true vite build) + cap sync

# Open the Android project in Android Studio to run on a device/emulator:
npm run cap:open:android  # = cap open android
```

`npm run build:native` on its own just produces a native-flavored `dist/`
(relative base, no service worker); `npm run cap:sync` is what you normally run,
since it also copies that `dist/` into `android/`. Re-run `cap:sync` after every
change to `src/` — the native project serves a *copy* of `dist/`, not the live
source.

Adding iOS later (on a Mac):

```bash
npm install @capacitor/ios
npx cap add ios
npm run cap:sync
npx cap open ios
```

## Why a separate build mode (`CAPACITOR_BUILD`)

The normal web build is tuned for GitHub Pages and differs from a native build
in two ways that would otherwise break the shell:

1. **Base path.** The Pages build uses `base: '/Gravy/'` (the repo subpath). A
   native shell serves assets from a local origin at the root, so a `/Gravy/`
   prefix would 404 every asset. Native mode uses a **relative base (`'./'`)**.
2. **Service worker.** The PWA service worker (vite-plugin-pwa) makes sense for a
   browser tab, but inside a Capacitor shell — which already serves the bundle
   locally and updates via the store / native OTA channel — a registered SW only
   fights that. Native mode **drops the PWA plugin entirely.**

Because dropping the plugin also removes the `virtual:pwa-register/react` module
that `src/components/UpdatePrompt.tsx` imports, native mode aliases that import
to an inert stub (`src/pwaRegisterStub.ts`) so the component still bundles and
becomes a no-op. Type-checking is unaffected — `tsc` resolves the import against
the real plugin types and doesn't follow the Vite alias.

## What's committed

The `android/` native project source is committed (standard Capacitor practice —
native code gets customized). Capacitor's generated `android/.gitignore`
excludes build artifacts and the copied web assets
(`android/app/src/main/assets/public`, generated `capacitor.config.json`), so
those are regenerated by `cap sync` rather than tracked.

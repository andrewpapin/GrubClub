# Capacitor native wrap (Epic 10 spike)

Reference for the Capacitor packaging spike — wrapping the existing Vite/React build in a
thin native shell so Gravy can ship to the App Store / Play Store and (later) reach native
capabilities (push, biometrics, haptics, camera). CLAUDE.md links here; read this when
touching the native build path. This is a **spike**: it proves the wrap works and pins the
toolchain, but stops short of signing, store config, and committed native customizations.

## Why Capacitor (not React Native / PWA-only)

Decided in `BACKLOG.md` Epic 10: Capacitor wraps `dist/` with near-zero rewrite of `src/`,
versus rewriting all 40+ components in `src/components/` for React Native. The same Vite/React
app keeps shipping to GitHub Pages as a PWA; the native shell is an additional target, not a
fork.

## What the spike added

- **Dependencies** — `@capacitor/core` (runtime), plus `@capacitor/cli`, `@capacitor/ios`,
  `@capacitor/android` (dev). Pinned to the `^8.x` line that was current at spike time.
- **`capacitor.config.ts`** (repo root) — `appId: com.gravyapp.app`, `appName: Gravy`,
  `webDir: dist`.
- **A native build mode** — `npm run build:native` runs `vite build --mode capacitor`, which
  forces a **root-relative base** (see below). `npm run cap:sync` chains `build:native` →
  `cap sync`. `npm run cap:open:ios` / `cap:open:android` open the native IDEs.

## The base-path gotcha (the one thing that will bite you)

`vite.config.ts` sets `base` to `/Gravy/` for a normal `build` because GitHub Pages serves the
app from the `andrewpapin.github.io/Gravy/` sub-path. A Capacitor WebView instead serves the
bundled `dist/` from the **root** of a custom scheme (`capacitor://localhost` on iOS,
`https://localhost` on Android), so a `/Gravy/`-prefixed asset URL 404s every script and
stylesheet — you get a blank white screen.

The fix is mode-driven: `--mode capacitor` selects `base: '/'`. Always build the native bundle
with `npm run build:native` (or `cap:sync`), never plain `npm run build`. Verified: the native
bundle emits `src="/assets/..."` while the Pages bundle still emits `src="/Gravy/assets/..."`.

## Local workflow

```bash
npx cap add ios            # one-time: scaffold ios/ (uses Swift Package Manager, no CocoaPods)
npx cap add android        # one-time: scaffold android/ (Gradle template, no Android Studio needed)
npm run cap:sync           # build:native + copy dist/ into both native projects
npm run cap:open:ios       # open Xcode  (needs macOS + Xcode to build/run)
npm run cap:open:android   # open Android Studio (or `cd android && ./gradlew assembleDebug`)
```

## Why `ios/` and `android/` are gitignored

The native projects are generated by `npx cap add` and are fully regenerable from
`capacitor.config.ts` + `dist/`. They also embed a copy of the built web bundle
(`android/app/src/main/assets/public`, `ios/App/App/public`) and carry tooling-version churn
(Gradle wrapper, `.xcodeproj`), so committing them now would add ~3 MB of regenerable, noisy
artifacts to every diff. They are `.gitignore`d for the spike.

**Graduation step:** once the wrap moves past a spike and the shells start carrying real
customizations (`Info.plist`, `AndroidManifest.xml`, app icons/splash, signing config), commit
the native folders so those changes are tracked — at that point regenerability stops being the
point and the folders become source.

## Known follow-ups (not in this spike)

- **Service worker inside the WebView.** `vite-plugin-pwa` still registers a Workbox SW in the
  native bundle. Inside a Capacitor WebView that's redundant (the shell already bundles assets
  locally) and can cache-fight `UpdatePrompt`'s auto-reload. Decide whether to disable the PWA
  plugin under `--mode capacitor` before shipping a real build.
- **Signing, store config, native push (APNs/FCM), CI for native builds, OTA policy** — all
  tracked as separate Epic 10 items in `BACKLOG.md`; this spike is their prerequisite, not a
  substitute.
